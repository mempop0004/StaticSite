<p>콘텐츠 제작 시스템에서 이미지 단계가 자주 무너지는 이유는 이미지가 여전히 제작 구조 바깥에 남아 있기 때문이다. 글 구조와 메타는 규칙이 있는데, 이미지만 마지막 순간의 별도 요청으로 붙으면 속도와 품질이 같이 흔들린다.</p>

<p>후속 글은 <a href="/ko/units/blog-web-content-production-system/">웹 콘텐츠 제작 시스템 유닛 페이지</a>를 기준으로 이어가고, 이 글은 <a href="/ko/blog/web-content-production-system-rules-you-should-fix-first/">기준 글</a>과 같이 묶어서 운영하는 편이 좋다.</p>

<h2>1. 슬롯마다 역할을 고정한다</h2>
<p>가장 단순한 규칙만으로도 흐름이 달라진다. 001은 항상 대표 이미지, 002는 항상 본문 설명 이미지로 두면 자산 역할을 매번 다시 합의할 필요가 없어진다.</p>

<h2>2. 파일 매핑이 바로 보여야 한다</h2>
<p>좋은 슬롯 시스템은 글 파일, 이미지 폴더, 프롬프트, 생성 자산 경로가 한 줄로 이어진다. 이 연결이 선명할수록 교체와 검수가 빨라진다.</p>

<h2>3. 프롬프트는 슬롯 역할을 따라가야 한다</h2>
<p>대표 이미지와 본문 설명 이미지는 해결하는 문제가 다르다. 그래서 시작부터 다른 프롬프트 규칙을 가져야 한다.</p>
<ul>
  <li>001 대표: 한 장면, 한 메시지, 썸네일 가독성</li>
  <li>002 본문: 과정 보조, 비교, 구조 설명</li>
</ul>

<h2>4. 상태 단계가 보여야 병목이 줄어든다</h2>
<p>초안, 이미지 대기, 001 완료, 002 완료, 리뷰 완료, 발행 준비 같은 단계가 보이면 흐름이 단순해진다. 이미지 상태가 숨겨져 있을수록 마지막 병목이 커진다.</p>

<h2>5. 슬롯별 실제 예시를 하나씩 남긴다</h2>
<p>시스템은 규칙만으로 굴러가지 않는다. 001 예시 하나, 002 예시 하나를 남겨두면 다음 글부터는 기존 패턴을 복사해 훨씬 빨리 적용할 수 있다.</p>

<h2>무엇부터 시작할까</h2>
<p>처음에는 슬롯 두 개와 상태 규칙 하나면 충분하다. 글 파일, 001, 002, 생성 경로가 헷갈리지 않고 반복되기 시작하면 나머지 이미지 시스템도 훨씬 쉽게 확장된다.</p>
