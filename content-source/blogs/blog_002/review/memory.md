# Review Memory

- 코드 예시는 짧고 바로 복사 가능한 형태를 우선한다.
- 추상적 설명보다 실제 워크플로를 먼저 쓴다.
- 도구 장단점은 비교표로 정리한다.
- 도입부는 "왜 지금 읽어야 하는가"를 먼저 말한다.
- 각 섹션에는 최소 한 개의 실무 예시나 프롬프트 예시를 넣는다.
- 내부 링크가 부족한 초기 단계에서는 유닛 페이지라도 연결해 글이 고립되지 않게 한다.
- 입문형 pillar 글은 제목을 더 강하게 잡고, 첫 문단에서 잘못된 시작점을 먼저 끊어 준다.
- 대표 이미지 alt는 설명 복붙이 아니라 시각적 역할을 짧게 요약한다.
- 개발자용 이미지는 추상 컨셉 아트보다 자료형 다이어그램을 우선한다.
- 이미지 프롬프트는 `증상`, `원인 후보`, `검증 포인트`처럼 구조가 바로 보이게 쓴다.
- 다음 글부터는 분위기용 추상 프롬프트를 피하고, 본문 설명에 바로 붙는 정보형 이미지를 우선한다.
- 개발자용 이미지에 영어 라벨이나 큰 텍스트가 보이면 바로 품질이 떨어지므로 `no letters`를 더 강하게 본다.
- hero도 질문표나 종이 더미를 많이 쓰기보다, 구조를 줄이는 장면이 먼저 읽히게 단순화한다.
