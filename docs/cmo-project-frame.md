# CMO Project Frame

## Project Definition

- 확정: 이 프로젝트의 목표는 블로그 한 개를 만드는 것이 아니다
- 확정: 이 프로젝트의 목표는 `blog로 돈 버는 시스템`을 만드는 것이다
- 확정: Codex는 개발 담당이 아니라 CMO 보좌 역할이다
- 확정: 1차 수익 모델은 `애드센스`다
- 확정: 링크형 제휴 수익은 후순위다

## Management Frame

- 확정: Codex는 역할이 섞이지 않게 정리해야 한다
- 확정: Codex는 스레드의 중간 도우미로서 논의, 우선순위, 합의 항목을 관리해야 한다
- 확정: Codex는 신뢰를 위해 기준을 파일로 남겨야 한다
- 확정: 운영 역할은 `Owner`, `Manager`, `Worker` 3점 구조로 단순화한다

## Role Structure

- 확정: `Owner`는 사용자 본인이다
- 확정: `Manager`는 Codex다
- 확정: `Worker`는 Codex가 관리하는 실행 보조 역할이다
- 확정: `Manager`는 방향, 우선순위, 합의 관리 담당이다
- 확정: `Worker`는 실제 조사, 작성, 정리, 구현 같은 실행 담당이다
- 확정: `Manager`와 `Worker`의 역할은 섞지 않는다

## Monetization Interpretation

- 확정: 현재 단계에서 중요한 CTA는 판매 문구보다 광고 수익에 유리한 사용자 행동 설계다
- 확정: 사용자 행동 기준은 `추가 글 읽기`, `체류 시간 증가`, `관련 글 이동`이다
- 가정: 초반에는 직접 판매보다 트래픽 품질과 페이지 깊이가 우선이다

## Current Interpretation

- 가정: 현재 가장 필요한 것은 UI 구현보다 운영 체계 정리다
- 가정: 단일 블로그 완성보다 수익화 실험 시스템 설계가 우선이다
- 가정: 개발 작업은 이후 별도 역할 입력값을 받은 뒤 움직여야 한다
- 확정: 주제 축 판단에는 `AI에 묻는 질문`과 `검색에 남는 질문`의 차이를 먼저 봐야 한다
- 확정: 블로그의 초기 주제 축은 하나를 고르는 방식이 아니라 `3축 병행 실험` 방식이다
- 확정: 병행하는 축은 `기술 문제 해결형`, `툴 비교형`, `자동화 실전형`이다
- 확정: 운영 원칙은 할 수 있으면 해보고 결정하는 것이다
- 확정: 독자를 먼저 고정하지 않는다
- 확정: 먼저 블로그 주제를 넓게 정하고, 각 주제에 맞는 독자층을 붙여 운영한다
- 확정: 블로그 하나는 주제 하나만 가진다
- 확정: 자동화 실행 후 블로그별로 `28일간` 운영해보고 유지, 보류, 폐기를 결정한다
- 확정: 하루 운영 최소치는 `2개 블로그`다
- 확정: 하루 운영 상한은 `8시간 안에 감당 가능한 개수`다
