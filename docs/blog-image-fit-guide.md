# Blog Image Fit Guide

## Goal

- 블로그 타입별로 어떤 이미지가 맞는지 미리 정한다.
- 앞으로는 이미지 프롬프트를 글 타입에 맞게 시작부터 다르게 잡는다.

## Core Rule

- 이미지는 분위기를 만드는 장식이 아니라, 글 이해를 빠르게 만드는 구조여야 한다.
- 블로그 타입에 따라 필요한 시각 언어가 다르다.
- 기본 슬롯은 `대표 1 + 본문 1`이다.
- 추가 이미지는 예외이며, 글이 먼저 충분히 강해야 한다.

## Slot Rule

- `001`: 대표 이미지
- `002`: 본문 핵심 설명 이미지
- `003` 이상: 예외적으로만 사용

### 001 / 002 Separation Rule

- `001`과 `002`는 같은 주제를 설명하더라도 같은 장면처럼 보이면 실패다.
- `001`은 글 전체를 대표하는 `핵심 긴장` 또는 `핵심 대비`를 맡는다.
- `002`는 본문에서 독자가 가장 헷갈리는 `구조`, `분류`, `단계`, `관계` 중 하나만 설명한다.
- `001`은 hero composition이어야 하고, `002`는 explanation diagram이어야 한다.
- `001`이 체크리스트 보드처럼 보이면 안 된다.
- `002`가 썸네일용 컨셉 장면처럼 보이면 안 된다.
- 색감이 비슷할 수는 있어도, 하는 일이 같아 보이면 다시 써야 한다.
- `001`은 너무 옅거나 은은해서 썸네일에서 핵심 객체가 안 잡히면 실패다.
- `002`는 분류판이나 관계판처럼 읽혀야 하고, 분위기 이미지처럼 보이면 실패다.

### 추가 이미지 허용 조건

- 글 자체가 이미지 없이도 충분히 읽힌다
- 핵심 섹션이 2개 이상 분명하다
- 추가 이미지가 새로운 설명을 실제로 맡는다
- 같은 구조를 반복하지 않는다

### 추가 이미지 금지 신호

- 글이 짧거나 평평해서 중간을 채우고 싶다
- 이미지가 없어도 이해가 거의 그대로다
- 대표 이미지와 본문 이미지가 같은 말을 반복한다
- 본문 밀도보다 이미지 수가 먼저 늘어난다

## Type Guide

### 1. 일반 사용자 글

- 적합한 이미지:
  - 정리 전/후 구조
  - 입력이 결과물로 바뀌는 흐름
  - 복잡함이 간단한 구조로 정리되는 장면
- 덜 맞는 이미지:
  - 너무 기술적인 다이어그램
  - 지나치게 추상적인 컨셉 아트
- 핵심:
  - `변화`와 `정리`가 빨리 보여야 한다

### 2. 개발자 글

- 적합한 이미지:
  - 증상 / 가설 / 검증 구조
  - 코드 흐름, 테스트 포인트, 리뷰 포인트 같은 자료형 다이어그램
  - 실제 워크플로를 요약한 정보형 이미지
- 덜 맞는 이미지:
  - 감성 위주 컨셉 이미지
  - 추상적 긴장감만 남는 커버
- 핵심:
  - `구조`, `검증`, `실무 포인트`가 먼저 보여야 한다

### 3. 운영 글

- 적합한 이미지:
  - 점검 순서
  - 경로, 응답, 상태, 호스트 같은 운영 요소의 관계도
  - 문제 진단 순서를 설명하는 자료형 이미지
- 덜 맞는 이미지:
  - 일반적인 서버실, 노트북, 사람 장면
  - 분위기만 전달하는 운영 컨셉 컷
- 핵심:
  - `무엇을 어떤 순서로 볼지`가 보여야 한다

### 검색 운영 글 추가 기준

- Search Console, CTR, 노출, 스니펫 같은 주제를 다룰 때는 `추상 신호 흐름`으로 가면 쉽게 공허해진다.
- 검색 운영 글 이미지는 아래처럼 더 구체적인 장면으로 내려야 한다.
  - 검색 결과에서 무엇을 기대하게 만드는가
  - 랜딩 첫 화면이 그 기대를 어떻게 받는가
  - 기대와 실제가 어디서 어긋나는가
- 즉 `데이터가 흐른다`보다 `검색 결과 기대와 클릭 판단 장면`이 보여야 한다.

### 4. 비교 글

- 적합한 이미지:
  - split comparison
  - 두 선택지의 차이가 한 축으로 읽히는 구조
- 덜 맞는 이미지:
  - 비교인데 한쪽만 강조된 배너형 이미지
- 핵심:
  - `비교 축`이 한눈에 읽혀야 한다

## Prompt Rule

- 일반 사용자 글:
  - `organize`, `simplify`, `turn into`, `clear structure`
- 개발자 글:
  - `symptom`, `hypothesis`, `verification`, `review markers`, `workflow`
- 운영 글:
  - `diagnostic order`, `response path`, `validation`, `host`, `status`
- 비교 글:
  - `comparison axis`, `side-by-side`, `selection criteria`

## Prompt Upgrade Rule

- 추상 명사만 던지지 말고, 반드시 아래 네 가지를 같이 적는다.
  - 보여야 할 객체
  - 화면 안 배치
  - 금지 요소
  - 직관성 체크
- 그리고 `001`과 `002` 각각에 대해 서로 닮지 않게 하는 금지 조건도 같이 적는다.
- `flow`, `system`, `signal`, `automation`, `reporting` 같은 추상 단어만 쓰면 추상 그래픽으로 흐르기 쉽다.
- 대신 `로그 묶음 1개`, `카드 3개`, `왼쪽-오른쪽 배치`, `텍스트 없는 아이콘`, `한 장면 한 메시지`처럼 장면 단위로 내려야 한다.
- `no text`만으로 부족하면 `no letters, no pseudo-text, no ui labels, no dashboard typography`까지 같이 넣는다.
- 여기에 더해 `no title words, no result labels, no before-after words, no platform names`처럼 글자 종류를 더 직접적으로 막는 편이 좋다.

### Label Dependence Rule

- 이미지가 구조를 글자가 아니라 객체 관계로 설명해야 한다.
- 아래 같은 직접 라벨은 현재 기준에서 기본 실패 신호다.
  - `TITLE`, `SUMMARY`, `TAGS`
  - `Before`, `After`
  - `Platform A`, `Platform B`
  - `locale URL`, `Result 1`
- 구조 설명이 필요하면 라벨 대신 아래로 바꾼다.
  - 제목: 더 큰 상단 블록 1개
  - 설명: 중간 요약 블록 1개
  - 태그: 작은 묶음 칩 여러 개
  - before/after: 좌우 대비만 두고 단어는 제거
  - locale/canonical/redirect: 아이콘 종류와 경로 방향만 다르게 둔다

### Hard Cases

- 아래 글 타입은 이미지 난도가 높다.
  - 비교 글
  - 메타 시스템 글
  - canonical / locale / redirect 같은 URL 신호 글
- 이 경우 더 공격적으로 단순화해야 한다.
  - 객체 수를 줄인다
  - 텍스트 라벨을 완전히 금지한다
  - 관계는 1개나 2개 축만 남긴다
  - 001과 002가 같은 board처럼 보이지 않게 장면 방식을 아예 갈라놓는다

### Prompt Template Split

- `001 cover` 프롬프트에는 아래를 넣는다.
  - 이 글의 핵심 긴장은 무엇인가
  - 한 장면에서 어떤 대비가 보여야 하는가
  - `must not look like a checklist board`
  - `must not look like an explanatory diagram`

- `002 inline` 프롬프트에는 아래를 넣는다.
  - 본문에서 독자가 가장 헷갈리는 구조가 무엇인가
  - 그 구조를 어떤 객체와 배치로 설명할 것인가
  - `must not look like a hero cover`
  - `must not look like a mood image`

## Keep / Regenerate Rule

- 아래에 맞으면 `유지`
  - 글의 핵심 구조가 이미지에서도 보인다
  - 본문 설명과 직접 연결된다
  - 블로그 썸네일과 본문 둘 다에서 의미가 있다

- 아래에 맞으면 `재생성 후보`
  - 분위기만 있고 정보 구조가 없다
  - 글 타입과 이미지 타입이 맞지 않는다
  - 개발자/운영 글인데 추상 컨셉 이미지에 가깝다
  - 본문 설명보다 장식 역할이 더 크다
