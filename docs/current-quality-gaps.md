# Current Quality Gaps

## Goal

- 이미 발행했거나 작업 중인 글에서 반복되는 약점을 확인한다.
- 개별 글 문제를 넘어서, 다음 글부터 미리 차단할 공통 결함을 정리한다.

- `blog_001`: AI 음성 메모를 회의 요약으로 바꾸는 글
- `blog_002`: 레거시 코드를 AI로 읽는 질문 글
- `blog_003`: 정적 자산 누락 점검 글
- `blog_004`: 기술 블로그 발행 리듬 글
- `blog_005`: Netlify vs Cloudflare Pages 비교 글
- `blog_006`: Crawled - currently not indexed 해석 글
- `blog_007`: 리뷰 결과를 규칙으로 누적하는 자동화 글
- `blog_008`: 현실적인 발행 주기 글
- `blog_009`: 제목/설명/태그 메타 시스템 글
- `blog_010`: locale 경로와 canonical 충돌 글

## Repeating Strengths

- 주제가 비교적 선명하고 검색형 문제에 맞는다.
- 문제 해결형 톤은 유지되고 있다.
- 체크리스트, 프롬프트, 점검 순서 같은 즉시 사용 요소를 넣으려는 방향은 맞다.
- 대부분 글에서 2번 섹션이 실제 중심축 역할을 하며, 예전보다 글의 강약이 살아났다.
- 본문은 전반적으로 `설명`보다 `판단 기준`을 남기는 방향으로 개선됐다.
- 운영 글과 시스템 글에서 겉으로 보이는 문제와 실제 문제를 분리하는 능력이 좋아졌다.

## Repeating Gaps

### 1. 기준 수가 아직 부족한 글이 있다

- 좋은 글은 읽고 나서 최소 3개의 판단 기준이 남아야 한다.
- 일부 글은 방향은 맞지만 기준 수가 적어 밀도가 부족하게 느껴진다.
- 특히 운영 글은 `무엇을 먼저 확인하고 무엇을 나중에 볼지`가 더 촘촘해야 한다.

### 2. 예시가 더 구체적이어야 한다

- 원칙 설명은 맞지만, 실제 사례가 한 장면 더 내려오지 않는 경우가 있다.
- 예시는 “이런 경우가 있다”가 아니라 “이 로그, 이 경로, 이 응답이면 이 가설을 본다” 수준이어야 한다.

### 3. 내부 안내 문장이 본문을 잠식하기 쉽다

- 일부 글은 다른 글로 보내는 문장이 본문 안에 섞이기 쉽다.
- 이 문장은 독자용 판단 기준이 아니라 작업자용 연결 습관에 가깝다.
- 앞으로는 관련 글 안내를 보조 요소로만 남기고, 본문은 글 하나만으로 성립해야 한다.

### 4. 결론이 행동보다 요약으로 끝나는 경우가 있다

- 마지막 문단은 “요약”보다 “첫 행동”이 남아야 한다.
- 독자가 지금 당장 열어볼 URL, 적어볼 문장, 확인할 항목이 끝에 있어야 한다.

### 5. 이미지가 설명보다 분위기로 기울기 쉽다

- 특히 개발자 글은 추상적인 컨셉 이미지로 가면 정보 밀도가 떨어진다.
- 자료형 구조가 필요한 글에서는 `증상`, `원인 후보`, `검증 포인트` 같은 블록이 먼저 보여야 한다.

### 6. hero 이미지가 mockup이나 광고 패널처럼 흐르기 쉽다

- `broken vs fixed` 방향 자체는 맞아도, 카드·버튼·UI 텍스트가 많아지면 블로그 hero보다 프로모션 목업처럼 보인다.
- hero는 요소 수를 줄이고 핵심 대비만 남겨야 한다.
- 특히 운영/문제 해결 글은 장식 패널보다 `깨진 상태 1개 vs 정상 상태 1개`가 더 직접적이다.

### 7. KO 본문에 작업자식 영어 표현이 섞일 수 있다

- `deeper bug` 같은 표현이 본문에 남으면 글 톤이 흔들린다.
- 검색형 글은 실무 용어를 쓰더라도, 문장 전체는 한 언어의 리듬으로 정리돼야 한다.
- 발행 전에는 “이 문장이 실제 독자 글투인가”를 한 번 더 봐야 한다.

### 8. 이미지가 여전히 직접 텍스트에 과하게 의존한다

- 오늘 완료한 `post_005` 묶음의 가장 큰 공통 약점은 이미지 안 직접 텍스트였다.
- `TITLE`, `SUMMARY`, `TAGS`, `Before`, `After`, `Platform A/B`, `locale URL`, `Result 1` 같은 라벨이 반복적으로 들어갔다.
- 구조 방향은 맞아도, 글자에 기대는 순간 이미지 품질이 한 단계 낮아진다.

### 9. 001과 002의 역할 분리는 좋아졌지만 아직 완전히 안정되진 않았다

- `blog_007`처럼 `001=핵심 대비`, `002=흐름 설명`이 잘 갈린 경우가 있었다.
- 반대로 여러 글에서는 둘 다 board나 system panel처럼 보여 역할이 겹치기 쉬웠다.
- hero가 `예쁜 정보 보드`로 흐르면 썸네일 힘이 약해지고, inline과 차별점도 사라진다.

### 10. 비교/메타/SEO 구조 글일수록 이미지 설계 난도가 높다

- 비교 글, 메타 시스템 글, canonical/locale 글은 글 자체는 잘 썼는데 이미지가 가장 자주 미달이었다.
- 이런 글은 구조를 설명하려다 이미지 모델이 곧바로 라벨과 도표 텍스트에 기대기 쉽다.
- 이 계열은 앞으로 `구조는 더 단순하게`, `객체는 더 적게`, `텍스트는 완전 금지`가 필요하다.

## Immediate Rules

- 모든 글은 본문 안에 판단 기준 3개 이상을 남긴다.
- 모든 글은 실제 장면 예시를 최소 2개 넣는 방향으로 쓴다.
- 내부 링크 유도 문장은 보조로만 쓰고, 본문 안에서는 최소화한다.
- 마지막 문단은 항상 첫 행동 하나로 끝낸다.
- 자료형 이미지가 필요한 글은 컨셉 아트보다 정보 구조를 우선한다.
- hero 이미지는 텍스트 많은 UI 목업보다 `핵심 대비 1개`가 먼저 보이게 줄인다.
- KO/EN 본문은 작업 중 섞인 어색한 혼합 표현이 없는지 마지막에 한 번 더 읽는다.
- 이미지 프롬프트에서 직접 라벨을 상상하게 하는 단어를 줄이고, 객체와 관계만 남긴다.
- 비교/메타/SEO 계열 이미지는 글자 없는 아이콘과 경로 구조만으로 읽히게 설계한다.
- `001`은 썸네일 대비, `002`는 설명 구조라는 역할 차이가 한눈에 안 보이면 다시 쓴다.
