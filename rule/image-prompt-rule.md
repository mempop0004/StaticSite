# Image Prompt Rule

- 이미지 프롬프트는 예시 문장 하나가 아니라 `슬롯 역할 가이드 + 개별 프롬프트` 구조로 쓴다.
- 슬롯 역할 가이드는 공용 기준을 먼저 주고, 개별 프롬프트는 해당 글의 실제 맥락만 추가한다.
- 슬롯 역할은 최소 `cover`, `inline`, `comparison`, `screenshot` 기준으로 나눈다.
- 우리는 `사진을 찍는 프롬프트`를 만드는 것이 아니다.
- 목표는 `해당 섹션 이해에 도움이 되는 블로그용 이미지`를 만드는 것이다.
- 따라서 프롬프트는 먼저 `어떤 시각 결과물이 필요한지`를 정해야 한다.
  - 예: 에디토리얼 커버
  - 예: 정보 중심 UI 컴포지트
  - 예: before/after 비교 이미지
  - 예: 단계 설명용 다이어그램형 이미지
  - 예: 실제 캡처 대체용 재구성 화면
- `cover`는 글 전체를 대표하는 한 장면이어야 한다.
- `cover`는 본문 설명을 하면 안 된다.
- `cover`는 사용자가 해석하지 않아도 바로 읽히는 `즉시 인식형 대비`여야 한다.
- `cover`는 `한 장면, 한 대비, 한 결과`만 남겨야 한다.
- `cover`에 여러 기능, 여러 패널, 여러 단계 과정을 동시에 넣지 않는다.
- `cover`는 `복잡한 입력 -> 정리된 출력`처럼 구조가 한눈에 읽히는 설명형 구성을 우선한다.
- `cover`는 영화 같은 조명, 책상 위 소품 과다, 멀티 모니터, 발광 효과, 반으로 쪼갠 화면 같은 연출 중심 구도를 금지한다.
- `cover`는 `2번 슬롯 같은 단정한 카드/패널/노트 기반 설명형 이미지`를 기본값으로 본다.
- `inline`은 본문 설명을 직접 돕는 경우에만 쓴다.
- 과정 설명, 단계 설명, 비교 설명은 `inline`이나 `comparison`이 맡는다.
- `comparison`은 차이가 한 화면에서 읽혀야 할 때만 쓴다.
- `screenshot`은 실제 설정 화면이나 도구 맥락이 중요할 때 쓴다.
- 프롬프트에는 `이미지 역할`, `이미지 설명`, `공통 기준`, `개별 프롬프트`, `참고 검색어`를 넣는다.
- 프롬프트에는 가능하면 `출력 형식`도 포함한다.
  - 예: `editorial composite illustration`
  - 예: `information-led visual`
  - 예: `clean UI concept composition`
  - 예: `diagram-style explanatory image`
- 텍스트를 여러 줄 얹는 이미지는 피한다.
- 표가 더 나은 경우는 이미지 대신 표를 쓴다.
- 클릭만 유도하는 과장 컷, 가짜 자막, 값싼 합성 느낌은 금지한다.
- `사람이 책상에 앉아 있는 일반적인 라이프스타일 사진`으로 빠지지 않게 금지 조건을 명확히 적는다.
- 사람보다 `핵심 객체`, `화면`, `문서`, `흐름`, `비교 포인트`가 먼저 읽혀야 한다.
- 어수선한 책상, 현실감 과한 사진풍 배경, 코드 화면 과다 노출은 대표 이미지에서 우선 금지한다.
- 기본은 `locale-safe image`다.
- 이미지 안에 한글/영문 문장을 넣지 않고, 설명은 본문 밖에서 처리한다.
- 언어가 들어가면 locale별 2벌 관리가 필요해지므로 기본 금지다.
- 대표 이미지는 `왜 클릭해야 하는가`를 설명이 아니라 대비로 전달해야 한다.
- 즉, 대표 이미지는 `글 전체 설명`이 아니라 `핵심 인상`을 담당한다.
